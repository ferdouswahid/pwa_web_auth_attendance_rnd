PWA Attendance System — Technical Reference

Code-level documentation covering data storage, authentication flows, biometrics, geofencing, and state management. Intended for engineering review.

Angular 21 Standalone Dexie 4 / IndexedDB WebAuthn Biometrics Leaflet 1.9 Tailwind CSS v4 No Backend · No HTTP
🏗️
Architecture Overview

This is a fully client-side Progressive Web App. There is no backend server, no REST API, no GraphQL endpoint, and no WebSocket connection. All business logic, data persistence, authentication, and geofencing run entirely in the browser.

Key architectural constraint: The app must function completely offline. All data is stored locally in IndexedDB via Dexie.js. Authentication state is persisted in sessionStorage. The service worker (Angular PWA) caches all application assets for offline use.
Frontend Layer
  • Angular 21 standalone components
  • Lazy-loaded routes
  • Reactive forms with validation
  • Angular Signals for state
Persistence Layer
  • IndexedDB (browser native)
  • Dexie.js ORM wrapper
  • 3 tables: users, attendance, zones
  • sessionStorage for auth state
Security Layer
  • SHA-256 password hashing (Web Crypto API)
  • WebAuthn platform authenticator
  • Functional route guards
  • No plaintext credentials stored
Platform APIs Used
  • crypto.subtle.digest — password hashing
  • navigator.credentials — biometrics
  • navigator.geolocation — location
  • indexedDB — data storage

Database — IndexedDB via Dexie.js

The app uses a single IndexedDB database named AttendanceDB managed by a single DbService class that extends Dexie. This is the only database in the application — all persistence goes through this one instance.

DbService — Schema Definition

@Injectable({ providedIn: 'root' })
export class DbService extends Dexie {
  users!:      Table<User,            number>;
  attendance!: Table<AttendanceRecord, number>;
  zones!:      Table<Zone,            number>;

  constructor() {
    super('AttendanceDB');

    // Version 1 — initial schema
    this.version(1).stores({
      users:      '++id, username',
      attendance: '++id, userId, timestamp',
    });

    // Version 2 — adds zones table; existing data preserved
    this.version(2).stores({
      zones: '++id, name',
    });
  }
}

Schema Key Syntax (Dexie)

NotationMeaning
++idAuto-incrementing integer primary key
usernameIndexed column (enables .where('username') queries)
userId, timestampMulti-index on attendance table for efficient user lookups
Versioning rule: When adding a new table or column, increment the version number and only list the new or changed stores in that version's .stores() call. Dexie merges all versions automatically and does not drop existing data.

Registration Flow

User self-registration is handled by RegisterComponent → AuthService.register(). It uses Angular Reactive Forms with a cross-field password-match validator.

1
User fills the registration form
Fields: username (3–30 chars), password (min 6 chars), confirmPassword. A custom passwordMatchValidator cross-validates the two password fields at the form group level.
form = fb.group({ username: ['', [required, minLength(3), maxLength(30)]], password: ['', [required, minLength(6)]], confirmPassword: ['', [required]], }, { validators: passwordMatchValidator });
2
Duplicate username check
Before inserting, AuthService queries Dexie for any existing record with the same username (case-insensitive via equalsIgnoreCase). If found, throws 'Username is already taken'.
const existing = await this.db.users .where('username').equalsIgnoreCase(username).first(); if (existing) throw new Error('Username is already taken');
3
Password hashing with SHA-256
The plaintext password is never stored. It is hashed using the browser's native crypto.subtle.digest('SHA-256', ...). The result is a 32-byte ArrayBuffer converted to a 64-character lowercase hex string.
const data = new TextEncoder().encode(password); const hashBuffer = await crypto.subtle.digest('SHA-256', data); const passwordHash = Array.from(new Uint8Array(hashBuffer)) .map(b => b.toString(16).padStart(2, '0')) .join(''); // → 64-char hex string
4
Write user to IndexedDB
db.users.add({ username, passwordHash }) inserts the record. Dexie auto-assigns id (auto-increment). WebAuthn fields (credentialId, publicKey, etc.) and zoneIds are absent at this point — they are populated on biometric enrolment and zone assignment.
await this.db.users.add({ username, passwordHash }); // Result stored in IndexedDB → AttendanceDB → users table
5
Success → redirect to /login
Component sets success = signal(true) and navigates to /login after 1.5 s. The user is not automatically logged in after registration.

Login Flow

Handled by LoginComponent → AuthService.login(). Credentials are verified by comparing SHA-256 hashes — the plaintext password is never sent anywhere and never persisted.

1
Look up user by username (case-insensitive)
const user = await this.db.users .where('username').equalsIgnoreCase(username).first(); if (!user) throw new Error('User not found');
2
Hash the submitted password and compare
The submitted password is hashed with the same SHA-256 routine used at registration. The resulting hex string is compared directly to user.passwordHash stored in IndexedDB.
const hash = await this.hashPassword(password); if (hash !== user.passwordHash) throw new Error('Invalid password');
3
Persist session to sessionStorage
On successful match, the full User object (including credentialId, WebAuthn fields, if set) is serialised as JSON and stored in sessionStorage under the key 'currentUser'. It is not stored in localStorage — the session ends when the tab or browser is closed.
// private persist(user: User): void this._currentUser.set(user); sessionStorage.setItem('currentUser', JSON.stringify(user));
4
Angular Signal updated — UI reacts instantly
_currentUser is a writable signal<User | null>. Components and guards read currentUser() (the readonly projection) — no subscriptions, no observables.
5
Navigate to /dashboard
Router navigates to /dashboard. The authGuard on that route now passes because auth.currentUser() is truthy.

Session Restoration on Page Reload

AuthService constructor reads sessionStorage.getItem('currentUser') on service initialisation (which happens at app bootstrap). If valid JSON is found, it is parsed and the signal is pre-populated — the user stays logged in within the same browser tab. 

constructor(private db: DbService) {
  const saved = sessionStorage.getItem('currentUser');
  if (saved) {
    try { this._currentUser.set(JSON.parse(saved) as User); }
    catch { sessionStorage.removeItem('currentUser'); }
  }
}

Route Guards

Both guards are functional (plain arrow functions), not class-based. Defined in guards/auth.guard.ts.

// authGuard — blocks unauthenticated access, redirects to /login
export const authGuard = () => {
  const auth   = inject(AuthService);
  const router = inject(Router);
  return auth.currentUser() ? true : router.parseUrl('/login');
};

// noAuthGuard — blocks already-logged-in users, redirects to /dashboard
export const noAuthGuard = () => {
  const auth   = inject(AuthService);
  const router = inject(Router);
  return !auth.currentUser() ? true : router.parseUrl('/dashboard');
};

Employee Management

The /employees route provides admin-level CRUD for users. It uses EmployeeService which wraps DbService directly. Any logged-in user can access this route (no role differentiation in this R&D prototype).

Create Employee

1
Admin fills form: username, password, zone assignments
Zone selection is a multi-select toggle — each zone's ID is collected into a Set<number> held in a signal (selectedZoneIds).
2
Duplicate username check + SHA-256 hash
const existing = await this.db.users .where('username').equalsIgnoreCase(username).first(); if (existing) throw new Error('Username already taken'); const passwordHash = await this.hashPassword(password);
3
Write to IndexedDB with zone assignments
await this.db.users.add({ username, passwordHash, zoneIds }); // zoneIds: number[] — IDs of zones employee may punch from

Update Employee

Uses db.users.update(id, patch). Password update is optional — if the admin leaves the password field blank, passwordHash is not included in the patch object and the existing hash is preserved. Zone assignments are always overwritten. 

const update: Partial<User> = { username: data.username, zoneIds: data.zoneIds };
if (data.password) {
  update.passwordHash = await this.hashPassword(data.password);
}
await this.db.users.update(id, update);

Delete Employee

db.users.delete(id) removes the user record from IndexedDB. Note: Attendance records for the deleted user are not cascaded/removed (they remain in the attendance table orphaned). This is expected behaviour for this R&D prototype.

Listing Employees

db.users.orderBy('username').toArray() returns all users sorted alphabetically. The component then maps each user's zoneIds to full Zone objects for display.

Zone Management

Geofence zones are circular areas defined by a centre point (lat/lng) and a radius in metres. The /zones page uses ZoneService for CRUD and renders each zone as a Leaflet circle on the map.

Haversine Distance Algorithm

Zone membership is determined in ZoneService.haversineDistance(). This calculates the great-circle distance between two GPS coordinates on the Earth's surface using the Haversine formula. No third-party geo library is used. 

haversineDistance(lat1: number, lng1: number, lat2: number, lng2: number): number {
  const R = 6371000;  // Earth radius in metres
  const toRad = (deg: number) => (deg * Math.PI) / 180;
  const dLat = toRad(lat2 - lat1);
  const dLng = toRad(lng2 - lng1);
  const a =
    Math.sin(dLat / 2) ** 2 +
    Math.cos(toRad(lat1)) * Math.cos(toRad(lat2)) * Math.sin(dLng / 2) ** 2;
  return R * 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1 - a));
}

The punch is only allowed when the returned distance is ≤ zone.radius: 

// checkUserZone: returns the first zone the user is currently inside
const active = zones.find(z =>
  this.haversineDistance(lat, lng, z.lat, z.lng) <= z.radius
) ?? null;

Zone–Employee Assignment

When an employee has zoneIds: [2, 5], the Punch page filters the zone list to only those IDs. If zoneIds is empty or undefined, the employee can punch from any zone. 

const userZoneIds = new Set(this.user()?.zoneIds ?? []);
this.zones.set(
  userZoneIds.size > 0
    ? allZones.filter(z => z.id !== undefined && userZoneIds.has(z.id))
    : allZones,  // fallback: all zones if none assigned
);

Biometric Authentication (WebAuthn)

Biometric login is implemented via the Web Authentication API (WebAuthn Level 2) using a platform authenticator — this means the device's built-in biometric sensor (fingerprint reader, Face ID, Windows Hello, etc.) is used, not a hardware security key. No biometric data is ever transmitted or stored — the browser/OS handles all biometric matching internally and communicates only credential identifiers to the app.

Support check: BiometricService.isSupported() verifies window.PublicKeyCredential exists before any WebAuthn call. If not available, biometric buttons are hidden and the user must use password-only punch.

Registration — registerBiometric(username)

Called from the Punch page when the user clicks "Enable Fingerprint". Returns a BiometricRegistration object with all credential data to persist.

export interface BiometricRegistration {
  credentialId:       string;         // base64 rawId
  webAuthnUserId:     string;         // hex random bytes used as user.id
  attestationObject:  string;         // base64 CBOR
  publicKey:          string | null;  // base64 DER/SPKI — null if browser unsupported
  publicKeyAlgorithm: string;         // e.g. 'ES256 (ECDSA P-256)'
  origin:             string;
  registeredAt:       number;         // Date.now()
}
1
Generate cryptographically random challenge and userId bytes
const challenge = crypto.getRandomValues(new Uint8Array(32)); const userIdBytes = crypto.getRandomValues(new Uint8Array(16));
2
Call navigator.credentials.create()
Browser invokes the platform authenticator. The user is prompted for their biometric (fingerprint/Face ID). The authenticator creates a new asymmetric key pair, stores the private key internally (on-device, never exported), and returns a PublicKeyCredential containing the public key and credential ID.
const credential = await navigator.credentials.create({ publicKey: { challenge, rp: { name: 'R&D Attendance System', id: window.location.hostname }, user: { id: userIdBytes, name: username, displayName: username }, pubKeyCredParams: [ { alg: -7, type: 'public-key' }, // ES256 (ECDSA P-256) { alg: -257, type: 'public-key' }, // RS256 (RSASSA-PKCS1-v1_5) ], authenticatorSelection: { authenticatorAttachment: 'platform', // device biometric only userVerification: 'required', // biometric MUST be verified residentKey: 'preferred', }, timeout: 60000, attestation: 'none', }, });
3
Extract public key via getPublicKey()
AuthenticatorAttestationResponse.getPublicKey() (Chrome 85+, Firefox 117+, Safari 16+) returns the SubjectPublicKeyInfo (SPKI) DER-encoded public key as an ArrayBuffer. getPublicKeyAlgorithm() returns the COSE integer which is mapped to a readable name.
const response = credential.response as AuthenticatorAttestationResponse; const pubKeyBuffer = response.getPublicKey?.() ?? null; const pubKeyBase64 = pubKeyBuffer ? btoa(String.fromCharCode(...new Uint8Array(pubKeyBuffer))) : null; const alg = response.getPublicKeyAlgorithm?.() ?? 0; const publicKeyAlgorithm = alg === -7 ? 'ES256 (ECDSA P-256)' : alg === -257 ? 'RS256 (RSASSA-PKCS1-v1_5)' : `COSE ${alg}`;
4
Return BiometricRegistration object to caller
return { credentialId: btoa(String.fromCharCode(...new Uint8Array(credential.rawId))), webAuthnUserId: [...userIdBytes].map(b => b.toString(16).padStart(2, '0')).join(''), attestationObject: btoa(String.fromCharCode(...new Uint8Array(response.attestationObject))), publicKey: pubKeyBase64, publicKeyAlgorithm, origin: clientData.origin, registeredAt: Date.now(), };
5
Save all fields to IndexedDB + refresh session
const reg = await this.biometricService.registerBiometric(user.username); await this.auth.updateCredentialId(user.id!, reg); // db.users.update(userId, { credentialId, webAuthnUserId, attestationObject, // publicKey, publicKeyAlgorithm, origin, registeredAt, // lastSignature: undefined, lastVerifiedAt: undefined }) // then re-reads user from DB and serialises to sessionStorage

Verification — verifyBiometric(credentialId)

Called every time the user punches in or out. Returns a BiometricVerification object.

export interface BiometricVerification {
  verified:      boolean;  // true if authenticator accepted the biometric
  lastSignature: string;   // base64 assertion signature for audit trail
}
1
Decode stored credentialId back to ArrayBuffer
const rawId = Uint8Array.from(atob(credentialId), c => c.charCodeAt(0));
2
Call navigator.credentials.get() with the credential
The browser prompts the user for their biometric. The platform authenticator signs the challenge with the private key (which stays on-device). If the biometric matches, the authenticator returns a PublicKeyCredential assertion.
const assertion = await navigator.credentials.get({ publicKey: { challenge, rpId: window.location.hostname, allowCredentials: [{ id: rawId, type: 'public-key', transports: ['internal'] }], userVerification: 'required', timeout: 60000, }, });
3
Extract signature and return BiometricVerification
The assertion's response.signature is the authenticator's cryptographic signature over the challenge + authenticatorData. It is captured as base64 for audit trail storage. The method returns { verified: true, lastSignature } if the OS accepted the biometric, or { verified: false, lastSignature: '' } if the user cancelled or the biometric failed.
const response = assertion.response as AuthenticatorAssertionResponse; const lastSignature = btoa(String.fromCharCode( ...new Uint8Array(response.signature) )); return { verified: true, lastSignature };
R&D scope note: WebAuthn's full security model requires server-side verification of the assertion signature using the stored public key. In this frontend-only prototype, we verify that the OS/browser accepted the biometric and returned a credential object — we capture and store the lastSignature bytes but do not validate the cryptographic signature against the stored public key. For production, a backend would need to verify the authenticatorData + signature using the public key extracted at registration.

Punch Flow (Clock In / Clock Out)

The most complex flow in the app. PunchComponent coordinates four services: LocationService, ZoneService, BiometricService, and AttendanceService.

1
Component initialises — load zones and start GPS watch
On ngOnInit, all zones are loaded from Dexie. If the user has zoneIds assigned, the list is filtered to only those. Then watchPosition is started.
const allZones = await this.zoneService.loadZones(); const userZoneIds = new Set(this.user()?.zoneIds ?? []); this.zones.set(userZoneIds.size > 0 ? allZones.filter(z => userZoneIds.has(z.id!)) : allZones); this.startLocationWatch();
2
Continuous GPS updates via watchPosition
navigator.geolocation.watchPosition fires on every location update. Each update: stores coordinates in coords signal, then calls updateZoneStatus() to compute the active zone.
this.watchId = this.locationService.watchPosition( coords => { this.coords.set(coords); this.updateZoneStatus(coords); // recomputes activeZone signal }, error => this.locationError.set(error.message), );
3
Zone membership check on each GPS update
updateZoneStatus() iterates all permitted zones and calls haversineDistance() for each. The first zone where distance ≤ radius becomes the activeZone. The nearestZone is always updated for UI display.
4
User taps Punch In or Punch Out
Button is disabled unless all three are true: user is set, coords is non-null, and activeZone is non-null (user is inside a valid zone).
5
Biometric verification gate
If user.credentialId is absent, an error is shown requiring biometric setup first. Otherwise, verifyBiometric(credentialId) is called — the OS prompts for fingerprint. On success, lastSignature and lastVerifiedAt are persisted to IndexedDB.
if (!user.credentialId) { this.actionError.set('Fingerprint authentication is required. Please enable it first.'); return; } const result = await this.biometricService.verifyBiometric(user.credentialId); if (!result.verified) { this.actionError.set('Biometric verification failed. Please try again.'); return; } await this.auth.updateLastVerified(user.id!, result.lastSignature);
6
Write attendance record to IndexedDB
await this.attendanceService.punchIn( user.id!, coords.latitude, coords.longitude, activeZone.name, // denormalized snapshot ); // Inserts: { userId, type: 'in', timestamp: new Date(), lat, lng, zoneName }
7
GPS watch cleared on component destroy
ngOnDestroy calls locationService.clearWatch(watchId) to stop battery drain when navigating away.

Computed Zone Distance Table

The UI shows a live-updating table of all permitted zones with their real-time distances. This is a computed() signal that re-runs whenever coords() or zones() change — no manual subscriptions needed. 

zoneDistances = computed(() => {
  const c = this.coords();
  return this.zones()
    .map(z => ({
      zone:     z,
      distance: c ? this.zoneService.haversineDistance(c.latitude, c.longitude, z.lat, z.lng) : null,
      isActive: c ? this.zoneService.haversineDistance(c.latitude, c.longitude, z.lat, z.lng) <= z.radius : false,
    }))
    .sort((a, b) => (a.distance ?? Infinity) - (b.distance ?? Infinity));
});

Attendance Records

Writing Records

AttendanceService.punchIn() and punchOut() are symmetric — both insert a new row into db.attendance with type: 'in' or type: 'out' respectively. They do not check for duplicate punches or enforce business rules like "must punch in before punching out" — this is left to UI-level logic.

Reading Records

// Get all records for a user, sorted newest first
async getHistory(userId: number): Promise<AttendanceRecord[]> {
  const records = await this.db.attendance
    .where('userId').equals(userId).toArray();
  return records.sort(
    (a, b) => new Date(b.timestamp).getTime() - new Date(a.timestamp).getTime()
  );
}

// Get the most recent punch record (used to show last action on UI)
async getLastPunch(userId: number): Promise<AttendanceRecord | undefined> {
  const records = await this.getHistory(userId);
  return records[0];
}

Export

exportJson(records) serialises an array of attendance records to a formatted JSON string using JSON.stringify(records, null, 2). The caller is responsible for triggering a browser download (Blob + anchor element).

PWA & Offline Capability

The app is configured as a PWA via @angular/service-worker. The service worker is managed by Angular's built-in SW infrastructure — no custom service worker is registered manually.

ngsw-config.json — Caching Strategy
  • All application shell assets (JS, CSS, HTML) are cached on install
  • The app continues to load and function with no network connection
  • All data operations (read/write) are against local IndexedDB — no network needed
  • The manifest.webmanifest enables "Add to Home Screen" / installability
The PWA service worker only activates in production builds (ng build). During development (ng serve) the SW is not active to allow hot-reload.

Password storage SHA-256 hex digest via crypto.subtle. Plaintext never persisted.
Auth state storage sessionStorage only — cleared on tab close. Not localStorage.
Biometric data Zero biometric data stored by app. OS handles matching; app stores only credential ID (a reference handle).
WebAuthn origin binding rpId = window.location.hostname — credentials are bound to the origin and cannot be replayed cross-origin.
Challenge randomness crypto.getRandomValues(new Uint8Array(32)) — cryptographically secure 256-bit challenge per WebAuthn operation.
Geolocation GPS coordinates are only used for zone membership checks and are persisted as part of the attendance record. No external transmission.
No server communication There is no network traffic containing user data — no XSS payload exfiltration risk to a server.
R&D limitations No role-based access control. No WebAuthn server-side signature verification. No IndexedDB encryption. Not production-ready.
